# Vercel Deployment Preparation

DeepRead will use two Vercel Hobby projects connected to the same Git repository:

```text
deepread-web  -> apps/web
deepread-api  -> apps/server
```

This document is preparation only. Do not deploy until the Phase 11.8 owner checklist is complete.

## Project Settings

| Setting | `deepread-web` | `deepread-api` |
| --- | --- | --- |
| Root Directory | `apps/web` | `apps/server` |
| Framework preset | Next.js | Express |
| Install command | `bun install --frozen-lockfile` | `bun install --frozen-lockfile` |
| Build command | `bun run build` | `bun run build` |
| Output directory | `.next` | Leave blank; Vercel builds `app.ts` as one Express Function |
| Node.js version | `22.x` | `22.x` |
| Include files outside Root Directory | Enabled | Enabled |

Both projects require the repository root workspace manifest, `bun.lock`, and shared package source. Enable **Include source files outside of the Root Directory in the Build Step** in both project settings. The web build consumes `packages/api` types plus `packages/env` and `packages/ui`; the API consumes `packages/api`, `packages/auth`, `packages/db`, `packages/env`, and `packages/config`.

The API entrypoint is `apps/server/app.ts`. It is the only recognized Express entrypoint filename in the project root or `src`. The API build creates `dist/vercel-app.mjs`, a Node.js bundle containing the complete internal workspace runtime graph, then verifies its import and `/health` route under Node. The entrypoint loads that bundle without calling `listen`. Vercel's Express framework support routes the deployment to that one app without custom rewrites; standalone `apps/server/src/standalone.ts` remains responsible for local `listen`, signals, and Prisma shutdown.

Keep the API Build Command as `bun run build`. The server tsdown configuration bundles the transitive `@deepread/api`, `@deepread/auth`, `@deepread/db`, and `@deepread/env` runtime sources, including generated Prisma client code. It does not rely on Node executing workspace `.ts` package exports. The build fails if the bundle retains an unresolved `@deepread/*/src/*.ts` import.

Prisma Client is generated by the database workspace `postinstall`. Generation and schema validation use a non-connecting schema-only fallback when `DIRECT_URL` is absent. No install, build, startup, or deploy command runs migrations or seeds.

## Environment Variables

Use placeholders only in documentation. Every environment change requires a new deployment.

### Web Project

| Variable | Preview | Production |
| --- | --- | --- |
| `API_UPSTREAM_URL` | Exact HTTPS URL of the paired preview API | Exact HTTPS production API origin |
| `NEXT_PUBLIC_SERVER_URL` | Exact HTTPS preview web origin | Exact HTTPS production web origin |

`API_UPSTREAM_URL` is server-only despite being configured on the web project. No database URL or auth secret belongs in the web project.

### API Project

| Variable | Preview | Production |
| --- | --- | --- |
| `DATABASE_URL` | Pooled URL for the isolated preview database target | Pooled URL for the production database |
| `BETTER_AUTH_SECRET` | Unique preview secret | Unique production secret |
| `BETTER_AUTH_URL` | Exact paired preview web origin | Exact production web origin |
| `CORS_ORIGIN` | Exact paired preview web origin | Exact production web origin |
| `NODE_ENV` | `production` | `production` |

`OPENALEX_API_KEY` is optional. Profiling flags remain optional and default to `false`. Do not configure `DIRECT_URL` in Vercel: migrations use the guarded local owner workflow in [`production-database.md`](production-database.md).

Credentialed tRPC and Better Auth requests use relative `/trpc/*` and `/api/auth/*` paths on the web origin. Next.js rewrites only those paths to `API_UPSTREAM_URL`. CORS and Better Auth trust only the exact web origin. Do not use `*`, `*.vercel.app`, or arbitrary origin reflection.

## Preview Pair

Use one deterministic preview pair for deployment rehearsal rather than enabling arbitrary branch origins.

1. Create one preview deployment for each project to obtain or assign stable preview aliases.
2. If URLs are not known before the first build, use explicit `example.invalid` HTTPS placeholders for that build only.
3. Set API `BETTER_AUTH_URL` and `CORS_ORIGIN` to the preview web URL.
4. Set web `API_UPSTREAM_URL` to the preview API URL and `NEXT_PUBLIC_SERVER_URL` to the preview web URL.
5. Redeploy the API, then redeploy the web project.
6. Verify `/health`, `/ready`, authentication cookies, and one tRPC query.

Repeat the same exact-origin setup with production domains. Never make CORS permissive to accommodate generated preview URLs.

## Health And Rollback

- `GET /health` verifies function liveness without querying PostgreSQL.
- `GET /ready` performs a two-second bounded database check and returns `503` with sanitized details on failure.
- Responses include `x-request-id`; unexpected failures log only correlated sanitized context.
- Roll back application code by promoting a previous healthy Vercel deployment.
- Database rollback remains separate: use a Supabase recovery point or a reviewed forward migration, never `db push` or `migrate reset`.

## Admin Pipeline Duration

Vercel Hobby Functions with Fluid compute currently allow up to 300 seconds. Keep the initial function maximum at 300 seconds and validate real production timings.

| Operation | MVP guidance |
| --- | --- |
| Metadata remediation | Small transaction; expected to complete comfortably. |
| Duplicate merge | Usually bounded to one candidate group; review relation counts and keep early production groups small. |
| Classification batch | Bounded concurrency is 8 and the application cap is 500; begin with 100-250 papers until production timings are known. |
| OpenAlex ingestion | Five external pages and concurrent database writes can approach request limits when OpenAlex is slow; begin with 100-125 papers. |

No queue or worker is required for initial controlled use. Larger or repeatedly slow ingestion/classification workloads need later architectural work rather than higher request limits.

## Phase 11.8 Owner Actions

1. Create `deepread-web` and `deepread-api` from the same repository.
2. Apply the project settings and enable source access outside each Root Directory.
3. Create the separate production Supabase project and run the guarded migrations manually.
4. Configure Preview and Production environment variables with exact paired origins.
5. Deploy the deterministic preview pair and complete health, auth, public, user, and admin smoke tests.
6. Deploy production, verify `/health` and `/ready`, then create the first production user and assign admin access manually.
7. Confirm the Supabase Data API, grants, backups, pooling, and secret-rotation checklist.
8. Retain the previous healthy Vercel deployment and database recovery point for rollback.
